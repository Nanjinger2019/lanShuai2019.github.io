这里记录一个关于hexo部署后图片的路径问题。

首先，默认的情况下，hexo g命令可以将内容打包生成public文件夹，这个public文件夹下的内容就是部署到github仓库中，在公网上进行访问内容。
如果原本的makedown文件中含有外部资源，如果不是hexo默认的文件夹，hexo会将其打包到public的一级子目录下。
这意味着你的文件路径存在巨大的隐患。
为了解决这个问题，我在_config.yml文件中将url设置为我的公网url(http://域名),但是部署之后，发现在首页上的图片可以显示，但是如果进入某一个具体博文中，图片就无法加载了。打开控制台，发现图片变为了相对路径，这个相对路径就是在_posts文件夹下编写makedown文件时添加图片的相对路径，那样的话找不到也就是理所当然的了。

总不能每生成一次，就把public下的html文件都改一遍把？那么，有没有一种行之有效的解决办法的？

在网上看了一些大佬们的方法，发现有一个hexo的图片管理插件十分有用，可以通过下面这条命令安装：

安装之后，每hexo new一个makedown文件，就会生成一个同名的文件夹。在编写该makedown文件的时候，直接将图片放到该文件夹下。方便又快捷。

之后，使用hexo g命令打包，博客相关的资源会放到生成的博客index.html同目录下，这样，便有效解决了图片路径问题

当你看到站点的外观与下图所示类似时即说明你已成功安装 NexT 主题。这是 NexT 默认的 Scheme —— Muse

选择 Scheme

Scheme 是 NexT 提供的一种特性，借助于 Scheme，NexT 为你提供多种不同的外观。同时，几乎所有的配置都可以 在 Scheme 之间共用。目前 NexT 支持三种 Scheme，他们是：

  • Muse - 默认 Scheme，这是 NexT 最初的版本，黑白主调，大量留白
  • Mist - Muse 的紧凑版本，整洁有序的单栏外观
  • Pisces - 双栏 Scheme，小家碧玉似的清新
    Scheme 的切换通过更改 主题配置文件，搜索scheme关键字。 你会看到有三行 scheme 的配置，将你需用启用的 scheme 前面注释 # 去除即可。

设置 菜单

菜单配置包括三个部分，第一是菜单项（名称和链接），第二是菜单项的显示文本，第三是菜单项对应的图标。 NexT 使用的是 Font Awesome 提供的图标， Font Awesome 提供了 600+ 的图标，可以满足绝大的多数的场景，同时无须担心在 Retina 屏幕下 图标模糊的问题。

设置 侧栏

默认情况下，侧栏仅在文章页面（拥有目录列表）时才显示，并放置于右侧位置。 可以通过修改 主题配置文件 中的 sidebar 字段来控制侧栏的行为。侧栏的设置包括两个部分，其一是侧栏的位置， 其二是侧栏显示的时机。
设置侧栏的位置，修改 sidebar.position 的值，支持的选项有：

  • left - 靠左放置
  • right - 靠右放置

    目前仅 Pisces Scheme 支持 position 配置。影响版本5.0.0及更低版本。

设置侧栏显示的时机，修改 sidebar.display 的值，支持的选项有：

  • post - 默认行为，在文章页面（拥有目录列表）时显示
  • always - 在所有页面中都显示
  • hide - 在所有页面中都隐藏（可以手动展开）
  • remove - 完全移除

设置 头像

编辑 主题配置文件， 修改字段 avatar， 值设置成头像的链接地址。其中，头像的链接地址可以是：

地址 值

站点内的地址 将头像放置主题目录下的 source/uploads/ （新建 uploads 目录若不存在)
配置为：avatar: /uploads/avatar.png
或者 放置在 source/images/ 目录下
配置为：avatar: /images/avatar.png /

编辑 站点配置文件， 设置 author 为你的昵称。

2019-09-02我的hexo小记(二)

发表于 2019-09-02 | | 阅读次数:

本文将记录我的hexo博客在github上的搭建过程。

准备工作

  • github账号，没有的话注册一个
  • 本地hexo项目
  • node.js、git相关知识
  • 已经注册并备案的域名

搭建博客

创建仓库

在github上新建一个仓库用于存放个人博客的代码和文章，一般命名为用户名.github.io

将仓库url + .git写入_config.yml文件中的deploy下的repo中，运行一下指令，将项目代码放到github中

在浏览器中输入’github用户名 + github.io`，如果可以看到博客首页，则证明博客已部署到github

绑定域名

如果不绑定域名的话，可以使用github的默认用户名.github.io来访问，但是如果想更有自己的个性，可以绑定到域名。
我的域名是www.geeklan.cn， 在腾讯云上购买，备案等待了小一个月时间。

1）在github项目中添加CNAME文件, 文件中填写不带www的域名，如我备案的域名是www.geeklan.cn，则在CNAME中填写的域名是geeklan.cn

2）在github中配置解析的域名地址。到仓库主页面，找到setting,然后进入setting选项卡，找到github Page下的custom domain,填入域名，保存。

3）域名解析，在域名控制台中（可通过购买域名的服务商官网进入，我使用的是腾讯云）添加域名解析

一般添加2个

一个主机记录为www，记录类型A,记录值即为你在命令行界面ping 用户名.github.io得到的IP地址

另一个主机记录为@,记录类型为’CNAME’，记录值是用户名.github.io
保存，生效时间一般为10分钟，

4）域名生效后，访问域名，看是否能得到和原本在 用户名.github.io下同样的效果，是则hexo博客部署成功。

小结

至此为止，我们已经完成了0到1的突破，拥有了自己的博客。如果说把个人博客看做一栋大厦，我们已经完成了大厦的建造，虽然目前来看就是个毛坯房，但至少让我们有了一个可以遮风避雨的地方，至于样式、主题、模块种种，以及今后的工作之路，乃至人生之路，我只能说：面包会有的，一切都会有的。

Hexo的目录结构

hexo脚手架搭建之后，目录如下所示

_config.yml

全局配置文件，网站的很多信息都在这里配置，诸如网站名称，副标题，描述，作者，语言，主题，部署等等参数。

package.json

hexo框架的参数和所依赖插件

scaffolds

scaffolds是“脚手架、骨架”的意思，当你新建一篇文章（hexo new ‘title’）的时候，hexo是根据这个目录下的文件进行构建的。基本不用关心。

source

这个目录很重要，新建的文章都是在保存在这个目录下的.
_posts 。需要新建的博文都放在 _posts 目录下。

_posts 目录下是一个个 markdown 文件。你应该可以看到一个 hello-world.md 的文件，文章就在这个文件中编辑。

_posts 目录下的md文件，会被编译成html文件，放到 public （此文件现在应该没有，因为你还没有编译过）文件夹下。

themes

网站主题目录，hexo有非常好的主题拓展，支持的主题也很丰富。该目录下，每一个子目录就是一个主题。

_config.yml文件

_config.yml 采用YAML语法格式。
具体配置可以参考官方文档，_config.yml 文件中的内容，并对主要参数做简单的介绍
